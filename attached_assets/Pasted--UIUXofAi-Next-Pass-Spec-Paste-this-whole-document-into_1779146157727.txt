## 1. Home — tappable tool badges

**File:** `src/pages/Home.tsx`

The `TOOL_BADGES` array currently renders 5 decorative `<span>` chips. Convert them to `<Link>` components.

- Map each badge to a `ToolId` from `src/lib/toolPref.ts`. **Drop ChatGPT** — it has no install-step support. Final list: Claude, Cursor, Lovable, Figma Make.
- Each chip links to `/library?tool=<id>` (e.g. `/library?tool=cursor`).
- Add a hover state: brighten the lime border from `${LIME}55` to `${LIME}88`, and lift `background` from `${LIME}10` to `${LIME}1A`. Use Tailwind's `transition-colors`.
- Keep the existing check icon, font, and sizing — this is a behavior change, not a visual one.
- Add `aria-label="Browse library filtered for <Tool>"` on each link.

Reword the label above the row from "works in" to **"Browse by tool"** — they're actions now, not badges.

## 3. Library — sticky filter strip

**Files:** `src/pages/Library.tsx`, `src/pages/LibraryType.tsx`

Add a sticky filter row at the top of the results grid on both views.

Filter groups (each on its own row, with a small `MONO` section label):

- **Project type:** All · Mobile apps · SaaS · E-commerce · Dashboards · Marketing sites · Design systems
- **Style:** All · Minimal · Enterprise · Bold · Playful · Accessible · Dark mode
- **Tool:** All · Claude · Cursor · Lovable · Figma Make

Visual:

- Selected chip: `INK` background, `INK_ON_LIGHT` text.
- Unselected chip: transparent with `1px solid ${BORDER}`, `SUB` text.
- Live result count on the right side of the filter container: `<n> results` in `MONO`, `MUTED` color.

Behavior:

- Read filter state from URL query params (`?tool=cursor&type=saas&style=minimal`). Use wouter's `useSearch` and `useLocation`.
- Update URL params on chip change; chips reflect URL state on mount.
- Filter the items array against selected filters before passing to the grid.
- When a filter is "All", that dimension is ignored.

Items currently lack explicit `projectType` and `style` fields. Either:
- (a) Add these as optional fields on the `Item` type and back-fill the existing items, OR
- (b) Derive them from the existing `tags` array.

Pick (a) if cleaner — it's more durable and testable.

**Do NOT create a separate `/discover` route.** This replaces guided discovery entirely.

## 4. Bundle detail — 7-section coverage breakdown

**Files:** `src/pages/BundleDetail.tsx`, `src/lib/items.ts`, `src/lib/bundles.ts`, new component `src/components/SectionCoverage.tsx`

### 4a. Extend the bundle data shape

In `src/lib/items.ts` (wherever `BundleItem.bundle` is typed), add an optional field:

```ts
  typography: 'strong' | 'thin' | 'missing';
  spacing: 'strong' | 'thin' | 'missing';
  elevation: 'strong' | 'thin' | 'missing';
  shapes: 'strong' | 'thin' | 'missing';
  components: 'strong' | 'thin' | 'missing';
  dosDonts: 'strong' | 'thin' | 'missing';

```

Back-fill all existing bundles in `src/lib/bundles.ts` with plausible values inferred from their content. Most rows should be `strong`; expect 1–2 `thin` per bundle, occasional `missing` for shapes or dosDonts.

### 4b. Build `src/components/SectionCoverage.tsx`

Renders 7 rows. Each row:

- Left: section label (text — "Colors", "Typography", "Spacing", "Elevation", "Shapes", "Components", "Dos & Don'ts").
- Right: a small horizontal bar (72px wide × 4px tall, rounded), followed by a one-word state label in `MONO` `MUTED`.

State styling:

- `strong` → bar full width, fill `LIME`, label `"strong"`.
- `thin` → bar 60% width, fill `PEACH`, label `"thin"`.
- `missing` → bar 20% width, fill `MUTED`, label `"missing"`.

Tight row spacing (~6px between rows). All rows align cleanly on the right edge.

### 4c. Wire into BundleDetail

In the right-side aside (currently a 2×2 stat grid), replace the layout with:

1. Big overall stat at the top: `{bundle.coverage}%` with `LIME` accent dot, label "overall coverage".
2. Below it, the `SectionCoverage` component fed from `bundle.sectionCoverage`.
3. Below that, a compact line: `<tokens> tokens · <components> components · last verified <updatedAgo>` in smaller `SUB` text.

Keep the palette strip and the forks counter as they are.

## 7. Install-success — "did this work?" pulse

**Where it goes:** the `showInstall` section inside `src/pages/BundleDetail.tsx` (this is where bundle install-success actually surfaces today; `CopySuccess.tsx` redirects to BundleDetail for bundles). Add the pulse below the existing copy/download cards in the install steps grid.

Visual — tight one-line layout:

```
Did this work for you?      [ Yes ]   [ Not quite ]
```

- Left: label in `SUB` color, normal sans-serif, ~13px.
- Right: two small buttons (~28px tall), rounded full, transparent with `1px solid ${BORDER}`, `SUB` text on hover lightens to `INK`.
- Sits below the install cards grid with a thin divider above (`border-t ${BORDER_SOFT}`) and ~24px top padding.

Behavior:

- On **Yes** click: write `{ bundleId, ts, result: 'yes' }` to `sessionStorage` under key `uiuxofai:pulse`. Replace the row with `Thanks ✓` (lime check, no buttons). No further UI.
- On **Not quite** click: expand inline to show 5 small chips in a row: **Colors · Typography · Spacing · Components · Generic**. Tapping any chip writes `{ bundleId, ts, result: 'no', reason: <chip> }` to sessionStorage and collapses the row to `Thanks ✓`.
- **No display of pulse data anywhere user-facing.** This is signal for the editorial dashboard only.
- Skippable. If the user navigates away or scrolls past, no consequence.
- Once the pulse is recorded for a given `bundleId` in this session, render the row as `Thanks ✓` immediately on revisit (don't ask again).
