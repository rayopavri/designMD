# designmd — Frontend Implementation Plan

**Version 1.0 · May 2026 · Frontend scope only**

> This plan covers everything Replit needs to build the designmd frontend: pages, components, design system, forms, auth UX, and the API client layer. Backend implementation (API routes, database, AI calls, email) is explicitly out of scope and handled separately. The frontend uses a typed mock API layer that will be swapped for real endpoints later — design it so that swap is a one-file change.

## 1. Scope

### In scope (build these)

- Every UI page and route
- All React components (layout, forms, data display)
- Tailwind v4 design system with custom theme
- Firebase Auth client SDK integration (sign-in modal, session detection, route guards)
- Form schemas with Zod validation
- Client-side search via Orama
- TypeScript types for every entity
- Typed API client wrapping all fetch calls
- Mock implementations of every API call so the frontend works without a backend
- Responsive design (mobile-first, primary breakpoints: 640px, 1024px, 1280px)
- Loading states, error states, empty states
- Toast notifications for user feedback
- Keyboard navigation and basic accessibility (WCAG 2.1 AA)

### Out of scope (don't build these)

- API route implementations (`/api/*` exists only as stubs that read from mock data)
- Database queries / Drizzle schema (mock data lives in JSON files)
- Anthropic, Gemini, Firecrawl integrations (mocked responses)
- Email sending (logged to console)
- Cloud Tasks, scheduled jobs (none of this exists in frontend)
- Firebase Admin SDK (server-side auth verification — not needed yet)
- Server-side rate limiting (will be added when backend is built)
- Real Orama index build pipeline (use a static JSON for now)

### How the swap will work later

Every API call in the app goes through `lib/api/*` functions. Each function has two implementations:

```
lib/api/bundles.ts
├── bundles.mock.ts    ← used when NEXT_PUBLIC_USE_MOCKS=true
└── bundles.real.ts    ← used when NEXT_PUBLIC_USE_MOCKS=false (when backend is ready)
```

A single env var flip switches the app from mocks to real API. Replit builds against mocks; backend gets built alongside; flip env var; deploy.

This is critical. designmd's audience is professional designers — the UI is the product's calling card. If it looks generic or AI-generated, you lose the audience in the first 3 seconds.

### Typography principles

- **Headings**: Inter (or Inter Display when available) at weights 600–700. Tight letter-spacing on display sizes (`-0.02em`).
- **Body**: Inter at 400, with `cv02/03/04/11` features enabled for designer-quality glyphs.
- **Mono**: JetBrains Mono for design.md code blocks and companion prompts.
- **Line height**: 1.5 for body, 1.2 for headings, 1.7 for long-form reading.
- **No more than 2 weights** active on a single screen.

### Component principles

1. **Generous whitespace.** Default vertical padding on sections is 64px (mobile) / 96px (desktop). Cards have 24px internal padding. Forms have 16px gap between fields.

2. **Borders, not shadows, for separation.** 1px borders in `--color-border`. Shadows reserved for elevated surfaces (modals, dropdowns, hover states on cards).

3. **Subtle hover, decisive active.** Hover = slight bg shift (3-5% darker), no scale or transform. Active/selected = stronger bg + border change.

4. **No gradients except in 3 places:** the palette strip on bundle cards, the brand color glow on hero, the category accent dot. Gradients elsewhere look AI-generated.

5. **No rounded-everything.** Match shape to function: `rounded-md` on inputs/buttons, `rounded-lg` on cards, `rounded-full` only on avatars/badges/tags.

6. **No glassmorphism.** No backdrop-blur on cards. Looks dated.

7. **Use the category color as accent**, not as the whole component color. A 4px top border on a bundle card in the category's color is enough.

### Inspiration references

What the UI should feel like, in order of importance:
- **Linear** — for the polish, spacing, and subtle motion
- **Vercel dashboard** — for the typography and information density
- **Stripe docs** — for the navigation patterns and content cards
- **Notion sidebar** — for the account/admin section navigation

What it should NOT feel like:
- **Generic shadcn demo** — too much rounded-2xl, too many gradients on every card
- **Material UI** — heavy and bureaucratic
- **Bootstrap** — outdated
- **Tailwind UI marketing components** — overwrought and not designer-coded

## 6. API client layer

This is the most important architectural decision in the frontend. Every server call goes through this layer. The frontend never calls `fetch` directly.

### Pattern

```typescript

import bundlesData from '@/mock-data/bundles.json';
import type { Bundle, BundleQuery, BundleListResponse } from '@/types';

// Simulated network delay
const delay = (ms = 400) => new Promise(r => setTimeout(r, ms));

export async function getBundles(query: BundleQuery = {}): Promise<BundleListResponse> {
  await delay();
  let items = bundlesData as Bundle[];
  
  if (query.category) {
    items = items.filter(b =>

    );
  }
  if (query.style?.length) {
    items = items.filter(b =>

    );
  }
  if (query.tool && query.tool !== 'all') {
```

```typescript
// lib/api/bundles.real.ts
import { apiFetch } from './client';
import type { Bundle, BundleQuery, BundleListResponse } from '@/types';

export async function getBundles(query: BundleQuery = {}): Promise<BundleListResponse> {
  const params = new URLSearchParams();
  if (query.category) params.set('category', query.category);
  if (query.style) query.style.forEach(s => params.append('style', s));
  if (query.tool) params.set('tool', query.tool);
  if (query.limit) params.set('limit', String(query.limit));
  if (query.offset) params.set('offset', String(query.offset));
  
  return apiFetch<BundleListResponse>(`/api/bundles?${params}`);
}

export async function getBundle(slug: string): Promise<Bundle | null> {
  try {
    return await apiFetch<Bundle>(`/api/bundles/${slug}`);
  } catch (err) {
    if (err.status === 404) return null;
    throw err;
  }
}
```

### Base fetcher

```typescript
// lib/api/client.ts
import { getAuth } from 'firebase/auth';

export class ApiError extends Error {
  constructor(public status: number, public message: string, public details?: any) {
    super(message);
  }
}

export async function apiFetch<T>(path: string, init: RequestInit = {}): Promise<T> {
  const auth = getAuth();
  const token = auth.currentUser ? await auth.currentUser.getIdToken() : null;
  
  const response = await fetch(path, {
    ...init,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...init.headers,
    },
  });
  
  if (!response.ok) {
    let details;
    try { details = await response.json(); } catch {}
    throw new ApiError(response.status, response.statusText, details);
  }
  
  return response.json();
}
```


### Auth hook

```typescript
// hooks/use-auth.ts
import { auth } from '@/lib/auth/firebase-client';
import { onAuthStateChanged, User as FirebaseUser } from 'firebase/auth';
import { useEffect, useState } from 'react';

export function useAuth() {
  const [firebaseUser, setFirebaseUser] = useState<FirebaseUser | null>(null);
  const [loading, setLoading] = useState(true);
  
  useEffect(() => {
    return onAuthStateChanged(auth, (user) => {
      setFirebaseUser(user);
      setLoading(false);
    });
  }, []);
  
  return { firebaseUser, loading, isAuthenticated: !!firebaseUser };
}
```

### Login modal

Three states:
1. **Provider selection**: "Continue with Google" / "Continue with email"
2. **Email input** (if user picks email): email field, "Send magic link" button
3. **Magic link sent**: "Check your email — we sent a sign-in link to [email]"

After successful sign-in:
- If returnTo URL param exists, redirect there
- Else, if first sign-in, redirect to `/welcome`
- Else, redirect to `/account`

### Protected routes

Use Next.js middleware:

```typescript
// middleware.ts
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

const PROTECTED_PATHS = ['/account', '/generate'];
const ADMIN_PATHS = ['/admin'];

export function middleware(request: NextRequest) {
  const path = request.nextUrl.pathname;
  // Note: actual auth check happens client-side via Firebase listener
  // Middleware just helps with the redirect on direct navigation
  // (more thorough server-side check happens when backend is wired up)
  
  // For now, the AuthGate component handles redirect to /login
  return NextResponse.next();
}
```

For the actual auth check, use the `AuthGate` component:

```typescript
// components/auth/auth-gate.tsx
'use client';
import { useAuth } from '@/hooks/use-auth';
import { useRouter, usePathname } from 'next/navigation';
import { useEffect } from 'react';

export function AuthGate({ children, requireEditor = false }: Props) {
  const { firebaseUser, loading } = useAuth();
  const router = useRouter();
  const pathname = usePathname();
  
  useEffect(() => {
    if (!loading && !firebaseUser) {
      router.push(`/login?returnTo=${encodeURIComponent(pathname)}`);
    }
  }, [loading, firebaseUser, router, pathname]);
  
  if (loading) return <LoadingSpinner />;
  if (!firebaseUser) return null;
  
  // requireEditor check happens against /api/me when backend is wired
  return <>{children}</>;
}
```

## 16. Quality bar

This is the most important section. The product is BY designers FOR designers — the UI is the calling card.

### Non-negotiables

1. **No generic AI-generated design.** No 24px rounded corners everywhere, no purple gradients on hero, no emoji icons for sections, no "we use AI" badges, no glassmorphism cards.

2. **Typography is sacred.** Real type hierarchy. No 5 different text sizes on one page. Heading weights consistent. Line lengths under 75 characters. Reading rhythm.

3. **Whitespace earns its keep.** Don't pad just to pad. But cards have breathing room. Sections have clear separation. Forms aren't cramped.

4. **Loading states match the destination.** A bundle card skeleton has the same dimensions and structure as a real card. A page header skeleton has the same height as the loaded version. No layout shift.

5. **Empty states are designed, not blank.** Every empty state has: an illustration or icon, a clear message, a CTA to do something productive.

6. **Errors are helpful, not scary.** "We couldn't load this bundle" + "Try again" button, not "Error 500: Internal Server Error".

7. **Mobile is not an afterthought.** Test every screen at 375px. Touch targets minimum 44x44px. Modals are bottom sheets on mobile, dialogs on desktop.

8. **Animation is subtle.** Hover states are 150ms ease. Page transitions are 200ms. Nothing bounces. Nothing has parallax.

9. **Copy is sharp.** No "Let's get you set up!" — say "Set up your account." No "Awesome!" — say nothing or say "Updated." Active voice. Specific verbs.

10. **The product is the design system.** Don't decorate. Don't add illustrations that aren't necessary. The default is restraint.

### What "good" looks like in practice

- Every component, before being added, should look like it came from Linear or Vercel — not from a Tailwind UI template.
- If you find yourself adding a third gradient to a page, stop. Remove one.
- If a form has more than 7 fields, look for what can be defaulted, hidden, or moved to a settings page.
- If a card has more than 6 elements (logo, title, description, palette, coverage, votes), simplify it.
- Read every error message aloud. If it sounds like a robot, rewrite it.

### Things that will lose the audience

- shadcn defaults left unstyled (the default looks like every other AI site)
- Generic icons used too broadly (a 🎨 emoji is not a design system icon)
- Page titles like "Welcome to designmd!" instead of just the H1 of what's on the page
- "Get started" CTAs when the user clearly hasn't seen anything yet
- Hero with a generic illustration of "person looking at screen"
- Stock photography
- Light grey on slightly lighter grey (failed contrast)
- Bold text used to emphasize 5 different things on the same page

## Final notes for Replit

1. **Build phases in order, ship each before starting the next.** Don't build all UIs in parallel. Phase F1 has to be polished before Phase F2 starts — the components from F1 are reused throughout, and getting them right early saves rework.

2. **Don't substitute the tech stack.** Drizzle is for backend. Use TanStack Query for server state. Use Zustand only for UI state. Don't add Redux. Don't add tRPC. Don't bring in a different form library.

3. **The mock API is real architecture, not a placeholder.** Build it with the same care as if it were a production API client. The swap to real APIs should be one env var.

4. **Match the schema exactly in TypeScript types.** Don't simplify. Don't omit fields you "don't need yet". The full type from the schema in Section 5 of the full implementation plan is what's in the database — match it.

5. **Read the design system section twice.** The most common failure mode for AI-built designer products is generic UI. Look at Linear's website. Look at Vercel's dashboard. The visual quality bar is "this could have been designed by a human who cares."

6. **No filler content.** If you don't have real content, show an empty state. Don't fill a section with lorem ipsum or generic stock content that the user has to figure out is fake.

7. **Test on mobile.** Every screen. Every interaction. The audience uses laptops, but their clients and stakeholders will share links on phones.
